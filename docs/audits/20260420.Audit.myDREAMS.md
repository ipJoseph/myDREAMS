# myDREAMS Platform: Technical Audit & Executive Review

**Prepared for:** Joseph "Eugy" Williams, CTO / Owner
**Date:** April 20, 2026
**Scope:** Full codebase audit across architecture, security, performance, scalability, and code quality

---

## 1. EXECUTIVE SUMMARY

myDREAMS is a **140,000-line codebase** (72K Python, 15K TypeScript, 49K HTML) that has evolved rapidly from a personal tool into a production-ready real estate platform. The core architecture is sound: clear separation between ingestion, storage, and surfaces; a unified database model; and a modern frontend. However, the rapid growth has created **structural debt** that will increasingly slow development and introduce risk as we approach public launch.

**The three biggest concerns, in order:**

1. **The God Class problem** -- `database.py` (7,318 lines, 163 methods) and `property-dashboard/app.py` (8,555 lines, 153 routes) are monoliths that resist testing, debugging, and safe modification.

2. **Inconsistent data access** -- 10 different `get_db()` implementations, raw SQL scattered across route handlers, and 200+ files still referencing SQLite patterns despite the PostgreSQL migration.

3. **No test safety net** -- 32 test functions covering ~12% of 270 route handlers. No CI/CD pipeline. Changes to the God Class risk silent regressions.

**The good news:** The foundations are right. PostgreSQL, Supabase Auth, the adapter pattern for integrations, the PhotoManager module, and the ListingService are all well-designed. The platform is functional and serving real data. The work ahead is consolidation, not reinvention.

## 3. ARCHITECTURE ASSESSMENT

### What's Working Well

**Unified data model.** One PostgreSQL database, one `listings` table, `mls_source` distinguishes origins. This was the right call and it's paying off.

**Adapter pattern.** `apps/integrations/_base/Adapter` with `AdapterResult` gives us clean vendor integration. FUB is the first implementation; this pattern scales to JustCall, ShowingTime, etc.

**PhotoManager module.** Centralized photo orchestration with per-MLS adapters, atomic writes, skip-on-failure. This replaced scattered code across 10+ files.

**pg_adapter.** The sqlite3-compatible PostgreSQL wrapper lets 163 database methods work without rewriting SQL. Pragmatic and effective as a migration bridge.

**Next.js public site.** Modern stack (Next.js 16, React 19, Tailwind v4, App Router). 22 components, 20 pages. Clean API client with SSR support.

### What Needs Work

**No clear framework for the Python backend.** Flask is used, but minimally. No ORM, no migration framework (Alembic), no task queue (Celery), no structured error handling. The 163-method God Class is the framework, and it's showing strain.

**Dashboard is a single file.** 8,555 lines, 153 routes, no blueprints. The API uses blueprints properly (6 route modules), but the dashboard never got the same treatment. Every change to the dashboard risks breaking something unrelated.

**Raw SQL in route handlers.** `admin.py`, `public.py`, and `public_writes.py` bypass `DREAMSDatabase` and execute SQL directly. This creates two parallel data access patterns: some routes use the abstraction, others don't.

**No CI/CD.** No GitHub Actions, no automated tests on push, no lint checks. Deployment is manual SSH + systemctl. This is the highest-leverage infrastructure gap.

| Finding | Severity | Count | Status |
|---------|----------|-------|--------|

| `next-auth` pinned to beta version (5.0.0-beta.30) | MEDIUM | 1 package | Open |
| `__pycache__` directories committed to git | LOW | 21 dirs, 84 files | Open |

### What's Secure
- No hardcoded API keys or credentials in source (all via `.env`)
- CORS properly configured with origin whitelist (not wildcard)
- SQL injection: parameterized queries throughout, sort columns whitelisted
- No `dangerouslySetInnerHTML` or `innerHTML` in React components
- BBO data leak protection is well-tested (17 dedicated tests)

## 5. DATABASE LAYER DEEP DIVE

### The God Class: `DREAMSDatabase` (163 methods)

This single class handles leads, properties, activities, scoring, pursuits, requirements, intake forms, IDX caching, contact snapshots, pipeline views, matching, and more. It violates the Single Responsibility Principle at industrial scale.

**Recommended decomposition:**

| New Service Class | Methods to Extract | Est. Methods |
|-------------------|--------------------|-------------|
| `ContactService` | All lead/contact CRUD, scoring, snapshots | ~35 |
| `PropertyService` | Property CRUD, change tracking, matching | ~25 |
| `ActivityService` | Events, communications, activity timeline | ~15 |
| `PursuitService` | Pursuits, pipeline, packages, collections | ~15 |
| `AnalyticsService` | Scoring, call lists, smart lists, reports | ~15 |
| `RequirementsService` | Intake forms, stated/behavioral prefs | ~10 |
| `SettingsService` | Settings, config, feature flags | ~5 |
| `DREAMSDatabase` (slim) | Connection management, migrations only | ~10 |

### Connection Inconsistency

10 separate `get_db()` implementations exist. 3 files still use hardcoded `sqlite3.connect()`: `buyer-workflow/app.py`, `automation/smart_collections.py`, `vendor-directory/app.py`.

### Unused Database Tables

7 tables created in schema but never queried by application code:
`power_hour_sessions`, `power_hour_dispositions`, `alert_log`, `automation_log`, `buyer_activity`, `contact_daily_activity` (scripts only), `market_snapshots` (scripts only)

## 6. PERFORMANCE & SCALABILITY

### Current Performance Risks

| Issue | Location | Impact |
|-------|----------|--------|
| `time.sleep(1)` in Flask route handlers | `public_writes.py:259,617`, `user.py:287` | Blocks all concurrent requests |
| 166 GB photos served from local disk | `/mnt/dreams-photos/` | No CDN, no HTTP cache headers |
| Single gunicorn workers (2 per app) | `deploy/gunicorn.conf.py` | Limited concurrent handling |

### What's Already Right
- PostgreSQL connection pooling via `ThreadedConnectionPool(2, 10)`
- 15+ indexes on the listings table
- Photo atomic writes (temp + rename)
- ISR caching on Next.js (15-min revalidation in prod)

## 8. LANGUAGE & FRAMEWORK EVALUATION

### Are we using the right language?

**Python: YES.** Right choice for data ingestion, MLS sync, and API layer. Flask's synchronous nature is fine at current scale.

**TypeScript/Next.js: YES.** Modern, well-supported, excellent for public site.

### Framework Gaps & Third-Party Recommendations

| Gap | Recommendation |
|-----|----------------|
| ORM / query building | SQLAlchemy Core (not full ORM) |
| Database migrations | Alembic |
| Background tasks | Celery + Redis |
| Error monitoring | Sentry |
| CDN for photos | Cloudflare |
| CI/CD | GitHub Actions |
| Data validation | Pydantic |
| API documentation | Flask-RESTX or Flasgger |

## 10. FINAL ASSESSMENT

myDREAMS is a capable, functional platform built at speed. The data model is sound. The MLS integrations work. The public site is modern and fast. The photo system is solid. The PostgreSQL migration is complete.

The primary risk is not in what the platform does, but in how the code is organized. The God Class and monolithic dashboard make every change riskier than it needs to be. The lack of CI/CD means we're relying entirely on manual verification.

**Bottom line:** The platform is launch-ready with P0 fixes. The structural refactoring can happen in parallel with feature work. The foundation is solid; we're cleaning up the scaffolding.
